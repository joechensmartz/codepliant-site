## 3. Data Handling Procedures

### 3.1 Collection

- Collect only the minimum data necessary for the stated purpose (data minimization)
- Obtain appropriate consent or establish lawful basis before processing
- Provide clear privacy notices at all data collection points
- Document the purpose and lawful basis for each data collection activity

### 3.2 Storage

- **Restricted data** must be encrypted at rest using AES-256 or equivalent
- **Confidential data** must be stored in access-controlled systems
- All personal data must be stored in approved locations only
- No personal data on local workstations, USB drives, or personal cloud storage
- Uploaded files must be scanned for malware before storage
- Storage buckets must not be publicly accessible by default

### 3.3 Transmission

- All data transmitted over networks must use TLS 1.2 or higher
- API keys and credentials must never be transmitted in URLs or query parameters
- Email containing personal data must use encryption where possible
- File transfers to third parties must use approved secure channels

### 3.4 Processing

- Process data only for the purposes documented in the Record of Processing Activities
- Implement pseudonymization or anonymization where feasible
- Log all access to Restricted and Confidential data
- AI models must not be trained on customer personal data without explicit consent
- AI processing outputs must be reviewed for unintended personal data exposure
- Analytics data must be anonymized or pseudonymized where possible
- User consent must be obtained before analytics tracking is enabled

## 4. Access Control

### 4.1 Principles

- **Least Privilege:** Grant the minimum access necessary for the role
- **Need-to-Know:** Access to personal data only when required for job function
- **Separation of Duties:** Critical operations require approval from multiple parties

### 4.2 Access Levels

| Role | Restricted Data | Confidential Data | Internal Data | Public Data |
|------|----------------|-------------------|---------------|-------------|
| Administrator | Read/Write (logged) | Read/Write | Read/Write | Read/Write |
| Developer | No access | Read (in production) | Read/Write | Read/Write |
| Support Staff | No access | Read (with justification) | Read | Read |
| Third-Party Contractor | No access | As per DPA only | As per contract | Read |

### 4.3 Access Management

- All access must be authenticated and authorized
- Access permissions must be reviewed quarterly
- Access must be revoked within 24 hours of role change or termination
- Shared accounts are prohibited for accessing personal data

## 5. Data Disposal

### 5.1 Retention Periods

Data must be disposed of when it is no longer needed for the stated purpose or when the retention period expires:

| Data Type | Maximum Retention | Disposal Method |
|-----------|------------------|-----------------|
| Payment/transaction records | 7 years (legal requirement) | Secure archival then deletion |
| Analytics data | 26 months | Automated purge |
| AI interaction data | 90 days | Automated purge |
| Uploaded files | Duration of account | Secure deletion on account closure |
| Backups containing personal data | 90 days after source deletion | Automated purge from backup rotation |

### 5.2 Disposal Methods

- **Electronic data:** Cryptographic erasure or overwriting with random data
- **Database records:** Secure deletion with verification from all replicas and caches
- **Backups:** Allowed to expire naturally from backup rotation (max 90 days)
- **Physical media:** Shredding or degaussing
- **Third-party data:** Written confirmation of deletion from sub-processors

### 5.3 Disposal Verification

- Maintain a log of all data disposal activities
- Verify deletion from all systems, including backups and caches
- Obtain written confirmation from third parties when data is deleted on their systems

## 6. Data Protection Incidents

Any suspected or actual data protection incident must be reported immediately:

1. **Report** the incident to the Data Protection Officer within 4 hours of discovery
2. **Assess** the severity, scope, and potential impact on data subjects
3. **Contain** the incident to prevent further data loss
4. **Notify** supervisory authorities within 72 hours if required (GDPR Art. 33)
5. **Notify** affected data subjects without undue delay if high risk (GDPR Art. 34)
6. **Document** the incident, response actions, and lessons learned

Refer to the Incident Response Plan (`INCIDENT_RESPONSE_PLAN.md`) for detailed procedures.
