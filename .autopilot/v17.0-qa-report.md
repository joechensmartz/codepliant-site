## 4. False Positive Analysis

### Recurring false positive patterns (24 total):

| Pattern | Count | Examples |
|---------|-------|---------|
| Utility/validation libraries detected as services | 7 | zod (x4), @tanstack/react-query (x2), i18next (x1) |
| Sentry over-counting (multiple SDK packages counted separately) | 6 | @sentry/nestjs, @sentry/nextjs, @sentry/react, @sentry/profiling-node |
| JS services triggered by env vars in non-JS projects | 4 | nodemailer from SMTP vars, ioredis from REDIS vars, @aws-sdk from AWS vars |
| Config/framework libraries | 4 | @t3-oss/env-nextjs, @trpc/server, @prisma/extension-accelerate, @cloudflare/workers-types |
| Cross-ecosystem env var bleed | 3 | @aws-sdk/client-s3 in PHP/Ruby from AWS env vars |

### Recommendations to improve precision:
1. **Deduplicate Sentry packages** -- count @sentry/* as a single "Sentry" service (would eliminate 6 FPs)
2. **Exclude utility libraries** -- zod, @tanstack/react-query, i18next, @trpc/server are not data-collecting services (would eliminate 7 FPs)
3. **Scope env var detections to matching ecosystem** -- SMTP_USERNAME should not trigger nodemailer in Ruby/Go projects (would eliminate 4 FPs)
4. **Exclude type-only packages** -- @cloudflare/workers-types has no runtime behavior (would eliminate 1 FP)

If all four were implemented, precision would rise to **120/120 = 100%** (no false positives remaining in this test set).
