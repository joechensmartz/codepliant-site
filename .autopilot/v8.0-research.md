## Scope (from roadmap)
- Production-ready GitHub Action with proper error handling
- PR comment bot — auto-comment when new services detected in a PR

## Current State

Codepliant already has a basic GitHub Action (`action.yml`) using a composite action pattern:
- Sets up Node 22, runs `npx codepliant scan --json`
- `action/entrypoint.sh` checks for missing compliance docs
- Inputs: `path`, `output-dir`, `fail-on-missing`
- No PR commenting, no badge generation, no pre-commit support

## 1. PR Comment Bot

### How it works
Use `@actions/github` (Octokit) to post comments on PRs when new services are detected.

### Implementation pattern
```ts
import * as github from '@actions/github';
import * as core from '@actions/core';

const octokit = github.getOctokit(core.getInput('github-token'));
const context = github.context;

// Post or update a comment (avoid spam on repeated pushes)
const COMMENT_HEADER = '<!-- codepliant-scan -->';

async function upsertComment(body: string) {
  const { data: comments } = await octokit.rest.issues.listComments({
    ...context.repo,
    issue_number: context.payload.pull_request!.number,
  });

  const existing = comments.find(c => c.body?.includes(COMMENT_HEADER));

  if (existing) {
    await octokit.rest.issues.updateComment({
      ...context.repo,
      comment_id: existing.id,
      body: COMMENT_HEADER + '\n' + body,
    });
  } else {
    await octokit.rest.issues.createComment({
      ...context.repo,
      issue_number: context.payload.pull_request!.number,
      body: COMMENT_HEADER + '\n' + body,
    });
  }
}
```

### Key decisions
- **Upsert pattern** (update existing comment vs. creating new): Every top Action (Codecov, Super-Linter) uses the "sticky comment" pattern with a hidden HTML marker (`<!-- codepliant-scan -->`) to find and update the same comment on each push. This prevents PR comment spam.
- **Permissions**: The action needs `pull-requests: write` permission. Document this in `action.yml`.
- **Token**: Accept `github-token` input, default to `${{ github.token }}`.
- **Trigger**: Run on `pull_request` events (opened, synchronize, reopened).

### What to show in the comment
- Summary table of detected services (new vs. existing)
- Missing compliance documents
- Badge showing compliance status
- Link to run `npx codepliant go` to fix

### Recommended approach for Codepliant
Since the current action is composite (shell-based), the cleanest path is to **convert to a JavaScript action** using `@actions/core` and `@actions/github`. This enables:
- Native PR commenting via Octokit
- Structured output parsing (no shell piping through Node)
- Better error handling with `core.setFailed()`

Alternative: keep composite action but add a separate step that runs a bundled JS script for the PR comment. This is messier but avoids a full rewrite.

**Recommendation**: Convert to a JavaScript action. It is the standard for tools that interact with the GitHub API.

## 2. Compliance Badge Generation

### Shields.io pattern
Shields.io is the standard for project badges. 1.6B+ images/month. Two approaches:

#### Option A: Static badge URL (simplest)
Generate a badge URL that users paste into their README:
```
![Compliance](https://img.shields.io/badge/codepliant-compliant-brightgreen)
![Compliance](https://img.shields.io/badge/codepliant-2_issues-orange)
![Compliance](https://img.shields.io/badge/codepliant-non--compliant-red)
```

#### Option B: Dynamic badge via JSON endpoint
Host a tiny JSON file (e.g., in the repo or via GitHub Pages) and use Shields.io's dynamic badge:
```
https://img.shields.io/endpoint?url=https://raw.githubusercontent.com/user/repo/main/.codepliant/badge.json
```

Badge JSON format:
```json
{
  "schemaVersion": 1,
  "label": "codepliant",
  "message": "compliant",
  "color": "brightgreen"
}
```

#### Option C: Self-generated SVG (no external dependency)
Generate the SVG locally using the same template Shields.io uses. This aligns with Codepliant's "zero network calls" philosophy.

```ts
function generateBadgeSVG(status: string, color: string): string {
  const labelWidth = 70;
  const msgWidth = status.length * 7 + 10;
  const totalWidth = labelWidth + msgWidth;
  return `<svg xmlns="http://www.w3.org/2000/svg" width="${totalWidth}" height="20">
    <linearGradient id="b" x2="0" y2="100%">
      <stop offset="0" stop-color="#bbb" stop-opacity=".1"/>
      <stop offset="1" stop-opacity=".1"/>
    </linearGradient>
    <clipPath id="a"><rect width="${totalWidth}" height="20" rx="3" fill="#fff"/></clipPath>
    <g clip-path="url(#a)">
      <rect width="${labelWidth}" height="20" fill="#555"/>
      <rect x="${labelWidth}" width="${msgWidth}" height="20" fill="${color}"/>
      <rect width="${totalWidth}" height="20" fill="url(#b)"/>
    </g>
    <g fill="#fff" text-anchor="middle" font-family="Verdana,Geneva,sans-serif" font-size="11">
      <text x="${labelWidth/2}" y="15" fill="#010101" fill-opacity=".3">codepliant</text>
      <text x="${labelWidth/2}" y="14">codepliant</text>
      <text x="${labelWidth + msgWidth/2}" y="15" fill="#010101" fill-opacity=".3">${status}</text>
      <text x="${labelWidth + msgWidth/2}" y="14">${status}</text>
    </g>
  </svg>`;
}
```

### Recommended approach for Codepliant
**Option C (self-generated SVG)** as the primary method, with Option B as an alternative for users who prefer dynamic badges. Reasons:
- Aligns with "zero network calls" principle
- CLI can output the badge file directly (`codepliant badge > .codepliant/badge.svg`)
- GitHub Action can commit the badge to the repo automatically
- No dependency on shields.io uptime

Badge states:
| Status | Color | Meaning |
|--------|-------|---------|
| compliant | `#4c1` (green) | All required docs present and up-to-date |
| 2 issues | `#fe7d37` (orange) | Some docs missing or outdated |
| non-compliant | `#e05d44` (red) | Critical docs missing |
| scanning... | `#9f9f9f` (grey) | Scan in progress |

## 3. Pre-commit Hook Integration

### Husky (most popular for Node.js)
- 30k+ GitHub stars, dominant in the JS ecosystem
- `npx husky init` sets up `.husky/` directory
- `prepare` script in package.json runs on `npm install` — automatic setup for all contributors
- v9+ simplified: hooks are plain shell scripts in `.husky/`

### Lefthook (polyglot alternative)
- Faster than Husky (parallel hook execution)
- Single `lefthook.yml` config file
- Better for polyglot projects (Go, Ruby, Python)
- Growing adoption but smaller community

### Implementation for Codepliant
Codepliant should not force a hook manager. Instead, provide:

1. **A CLI command** that runs the compliance check:
   ```bash
   npx codepliant check    # exits 1 if non-compliant
   ```

2. **Setup instructions** for both Husky and Lefthook:

   **Husky:**
   ```bash
   npx husky init
   echo "npx codepliant check" >> .husky/pre-commit
   ```

   **Lefthook:**
   ```yaml
   # lefthook.yml
   pre-commit:
     commands:
       compliance:
         run: npx codepliant check
   ```

3. **Optional: `codepliant init-hooks`** command that detects which hook manager the project uses and adds the hook automatically:
   ```ts
   // Detect hook manager
   if (existsSync('.husky')) {
     appendFileSync('.husky/pre-commit', '\nnpx codepliant check\n');
   } else if (existsSync('lefthook.yml')) {
     // Append to lefthook.yml
   } else {
     // Offer to install husky
   }
   ```

### Key consideration
The pre-commit hook should be **fast**. A full scan of a large project could take seconds. Options:
- Only scan changed files (use `git diff --cached --name-only`)
- Cache previous scan results and only re-scan on relevant file changes
- Make the hook optional / configurable via `.codepliantrc`

## 4. GitHub Marketplace Listing

### Requirements for listing
From GitHub's official docs:

**All listings must:**
- Have a relevant description of the application
- Specify a pricing plan (can be free)
- Include a valid privacy policy link
- Provide a support method (link or email)
- Include logo, feature card, and screenshot images
- Have working Terms of Service link

**For GitHub Actions specifically:**
- Actions are listed automatically when you add `branding` to `action.yml`
- No approval process needed for free Actions
- Must follow the [GitHub Marketplace Developer Agreement](https://docs.github.com/en/site-policy/github-terms/github-marketplace-developer-agreement)

**For paid apps:**
- Need 100+ installations first
- Organization must be a verified publisher
- Must handle purchase events via GitHub Marketplace API

### What Codepliant needs in action.yml
```yaml
name: "Codepliant Compliance Check"
description: "Scan your codebase and auto-generate compliance documents"
branding:
  icon: "shield"
  color: "blue"
```

The `branding` block is what makes the Action appear in the Marketplace with an icon.

### Marketplace listing best practices (from analyzing top Actions)
1. **Codecov** — Clear value prop in first line, usage example front and center, badge in README
2. **Super-Linter** — Feature list with supported languages, comparison table, "getting started" in 3 steps
3. **Snyk** — Security-focused messaging, integration with GitHub Security tab, SARIF output

**Common patterns across top 5 developer tool Actions:**
- One-liner usage example in the README
- Minimal required inputs (most things have smart defaults)
- Outputs that integrate with other Actions (SARIF, JSON, exit codes)
- PR-level feedback (comments, checks, annotations)
- Badge showing status
- `permissions` block documented clearly

### Recommended Marketplace strategy for Codepliant
1. Add `branding` to `action.yml` immediately (this is the gate to Marketplace visibility)
2. Keep the Action free (drives adoption, aligns with open-source ethos)
3. Focus the listing description on the differentiator: "compliance docs from actual code, not questionnaires"
4. Provide a one-liner usage example
5. Target v10.0 (Launch Preparation) for polished Marketplace presence

## 6. Implementation Plan for v8.0

### Priority order
1. **Convert to JavaScript action** — Foundation for everything else
2. **PR comment bot** — Highest user-facing value
3. **Badge generation** — Low effort, high visibility
4. **Pre-commit hook** — CLI command + docs
5. **Marketplace listing** — Add branding to action.yml

### New dependencies needed
```json
{
  "@actions/core": "^1.10.0",
  "@actions/github": "^6.0.0",
  "@vercel/ncc": "^0.38.0"
}
```
Use `@vercel/ncc` to bundle the action into a single file (standard practice — avoids shipping node_modules).

### File structure
```
action/
├── index.ts          # Action entry point
├── comment.ts        # PR comment upsert logic
├── badge.ts          # SVG badge generation
└── dist/
    └── index.js      # ncc-bundled output

action.yml            # Updated: JavaScript action, branding added
```

### action.yml (updated)
```yaml
name: "Codepliant Compliance Check"
description: "Scan your codebase for compliance requirements and auto-generate documents"
branding:
  icon: "shield"
  color: "blue"

inputs:
  path:
    description: "Path to the project to scan"
    default: "."
  output-dir:
    description: "Directory for compliance documents"
    default: "legal"
  fail-on-missing:
    description: "Fail if required compliance documents are missing"
    default: "false"
  comment-on-pr:
    description: "Post scan results as a PR comment"
    default: "true"
  github-token:
    description: "GitHub token for PR comments"
    default: "${{ github.token }}"
  badge:
    description: "Generate compliance badge SVG"
    default: "false"

outputs:
  status:
    description: "Compliance status (compliant, issues, non-compliant)"
  services-count:
    description: "Number of detected services"
  missing-docs:
    description: "Comma-separated list of missing documents"

runs:
  using: "node20"
  main: "action/dist/index.js"
```

### New CLI commands
```bash
codepliant check          # Exit 0 if compliant, exit 1 if not (for hooks/CI)
codepliant badge          # Output badge SVG to stdout
codepliant init-hooks     # Set up pre-commit hook (detects husky/lefthook)
```
