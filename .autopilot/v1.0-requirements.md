## 1. Scanner Accuracy Issues

### P0 - Must Fix

**1.1 Env scanner has dangerously loose matching (env.ts:37-42)**
The env var matching logic uses `v.includes(envPattern) || envPattern.includes(v)`. This means a user env var like `KEY` would match `OPENAI_API_KEY` (because `"OPENAI_API_KEY".includes("KEY")`), and an env var `STRIPE_SECRET_KEY_OLD` would match `STRIPE_SECRET_KEY` (because includes is substring). This will produce massive false positives. Any short env var name becomes a wildcard. The `envPattern.includes(v)` direction is particularly dangerous -- a user var named `API` would match every service that has "API" anywhere in its env patterns.

**1.2 Dependency scanner has wrong scoped package matching (dependencies.ts:22-24)**
The logic `dep.startsWith(sigName.split("/")[0] + "/")` means if the signature is `@sentry/node` and the user has `@sentry/nextjs`, it matches `@sentry/node` as the detected service. But the user doesn't have `@sentry/node` -- they have `@sentry/nextjs`. The test at dependencies.test.ts:79 explicitly tests this wrong behavior: asserting `@sentry/node` is detected when `@sentry/nextjs` is the actual dependency. The service name in the output is misleading.

**1.3 Import scanner only finds the first match per file (imports.ts:107-128)**
`jsImportRegex.exec(content)` is used without resetting `lastIndex`, and `||` short-circuits. If a file has both a Python-style import and a JS-style import of different services, the regex state could cause missed detections. Also, once any pattern for a signature matches, it `break`s out -- fine for one signature, but the sequential `||` across three language regexes could interact badly with `lastIndex` since the regexes have the `g` flag.

**1.4 Import scanner Python/Go regexes run on JS/TS files (imports.ts:84-108)**
Every file (including `.ts`, `.tsx`, `.js`) is tested against Python import regex and Go import regex. While unlikely to cause false positives in practice, it's wasteful and could theoretically match Go-style backtick strings in JS template literals.

### P1 - Should Fix

**1.5 No `.env.production` scanning (env.ts:9)**
The ENV_FILES list includes `.env`, `.env.example`, `.env.local`, `.env.development` but not `.env.production`, `.env.staging`, or `.env.test`. Many real projects use these.

**1.6 Python package name matching is case-sensitive in one direction (python.ts:308)**
`pkgName === sigName.toLowerCase() || pkgName === sigName` -- this means `SQLAlchemy` in PYTHON_SIGNATURES will match `sqlalchemy` (lowercased) but the reverse direction `pkgName === sigName` would only match exact case. This works for requirements.txt (lowercased on line 227) but Pipfile entries (line 293) are lowercased too, so `PyJWT` in PYTHON_SIGNATURES will match via `.toLowerCase()`. Fragile and confusing logic.

**1.7 pyproject.toml parsing is fragile (python.ts:246-265)**
Simple regex-based TOML parsing. The regex `dependencies\s*=\s*\[([\s\S]*?)\]` uses lazy matching and will break on nested brackets or multi-line values with brackets. Also, Poetry's `[tool.poetry.dependencies]` section regex captures everything until next `[` or end of string, which could accidentally grab non-dependency sections.

**1.8 Import scanner skips directories starting with `.` except explicitly listed ones (imports.ts:53)**
`!entry.name.startsWith(".")` means `.github/scripts/`, `.husky/`, etc. are skipped -- fine. But it also means `.src/` or any dot-prefixed source directory would be missed. Minor edge case but inconsistent with how Go/Python scanners work.

**1.9 Go module matching doesn't handle `// indirect` comments (golang.ts:172-176)**
The `go.mod` parser splits on whitespace to get the module path, which works, but indirect dependencies (marked with `// indirect`) are still matched. This could detect services that are transitive dependencies the project doesn't directly use, leading to false positives.

### P2 - Nice to Fix

**1.10 DATABASE_URL env pattern matches both prisma and drizzle (types.ts)**
Both `prisma` and `drizzle` signatures have `DATABASE_URL` as an env pattern. If a project has `DATABASE_URL` in `.env`, both will be detected. Should deduplicate or prioritize based on other evidence.

**1.11 No scanning for Ruby, PHP, Java, or Rust projects**
Only JS/TS, Python, and Go are supported. Not a bug but worth documenting as a known limitation.

## 2. Generated Document Accuracy Issues

### P0 - Must Fix

**2.1 Privacy Policy is missing 9 of 15 GDPR-required clauses**
Under GDPR Article 13/14, a privacy policy MUST include: (1) identity of controller, (2) DPO contact, (3) purposes of processing, (4) lawful basis for EACH purpose, (5) legitimate interest details, (6) recipients/categories of recipients, (7) international transfer details & safeguards, (8) retention periods per purpose, (9) data subject rights, (10) right to withdraw consent, (11) right to lodge complaint, (12) whether data provision is statutory/contractual, (13) existence of automated decision-making, (14) source of data if not from data subject, (15) any further processing intention. Current template covers maybe 6 of these partially. Missing: DPO, per-purpose lawful basis, legitimate interest balancing test, specific recipient disclosure, international transfer mechanism, retention per purpose (current is generic), withdrawal of consent, statutory/contractual obligation, automated decision-making, source of data.

**2.2 Terms of Service has literal `${jurisdiction}` in output (terms-of-service.ts:7)**
When no jurisdiction is configured, the fallback is the string literal `"${jurisdiction}"` -- a template literal syntax that got left as a raw string. Users see `${jurisdiction}` in their generated Terms of Service document. Should be a human-readable placeholder like `[Your Jurisdiction]`.

**2.3 Section numbering is fragile and breaks with combinations (terms-of-service.ts:82, 91-94)**
The ToS uses conditional section numbering with manual arithmetic (`hasAI && hasPayment ? "7" : hasAI || hasPayment ? "6" : "5"`). This is error-prone. If a new conditional section is added, every number after it must be manually recalculated. For example, with AI + Payment + Storage, the User Content section tries to compute its number but the logic only accounts for 2 flags, not the combinatorial explosion.

**2.4 Privacy Policy section numbering is also fragile (privacy-policy.ts:69-107)**
Uses `aiServices.length > 0 ? "5" : "4"` etc. for section numbers. If the third-party section or AI section is absent, the numbering shifts, but the logic only accounts for the AI conditional, not the third-party conditional.

**2.5 AI Disclosure risk classification is purely informational, provides no actual assessment (ai-disclosure.ts:48-57)**
The document just lists all four risk categories and tells the user to figure it out themselves. The tool has enough information to make at least a basic assessment (e.g., if no biometric/healthcare/employment services detected, it's likely Limited or Minimal risk).

### P1 - Should Fix

**2.6 Cookie Policy lists CSRF token for all projects (cookie-policy.ts:47)**
Every generated cookie policy includes "CSRF token" as an essential cookie, even when there's no evidence the app uses CSRF tokens. This is a made-up detail.

**2.7 Privacy Policy claims "We do not use your data to train AI models" (privacy-policy.ts:63)**
This is a blanket claim that may not be true. The tool has no way to verify this. Should say something like "Review your agreement with each AI provider regarding data training."

**2.8 DPA is a template addressed to no specific processor (dpa.ts)**
The DPA lists sub-processors but the agreement itself has no "Data Processor" party -- only a "Data Controller." A real DPA is a bilateral agreement between controller and processor. This template is more of an internal reference document than an actual DPA.

**2.9 AI Disclosure provider policy URLs may be stale (ai-disclosure.ts:130-139)**
Hardcoded URLs like `https://openai.com/policies/privacy-policy` or `https://www.anthropic.com/privacy` will break when these companies change their URL structure. No validation.

**2.10 Data retention periods are fabricated (privacy-policy.ts:70-75)**
"Usage data: Retained for up to 26 months" and "Account data: Retained until you delete your account" are made-up numbers with no basis in the scanned code. The tool has no way to determine actual retention periods.

### P2 - Nice to Fix

**2.11 No "last updated" tracking across regenerations**
Documents always use today's date. No way to see what changed between generations or maintain a history.

**2.12 Generated documents don't include table of contents**
For longer documents (Privacy Policy, DPA), a table of contents would improve usability.

## 3. Code Quality Issues

### P0 - Must Fix

**3.1 MCP server has runtime dependency on `@modelcontextprotocol/sdk` and `zod` (package.json:44-46)**
The CLAUDE.md says "No runtime dependencies -- only devDependencies" but `@modelcontextprotocol/sdk` is in `dependencies`. Also, the MCP server imports `zod` (via the SDK), which means the project actually has transitive runtime dependencies. The "zero runtime dependencies" claim is violated. Either move it to devDependencies with a separate build, or update the docs.

**3.2 `main()` in cli.ts is not async but calls async function without await (cli.ts:143)**
`runInit(absProjectPath).then(() => process.exit(0))` works but if the promise rejects, the error is silently swallowed. No `.catch()` handler. An error in the interactive init flow crashes silently.

**3.3 Config loading silently swallows malformed JSON (config.ts:27-32)**
`loadConfig` catches JSON.parse errors and returns defaults without any warning. A user with a typo in their config file gets zero feedback that their config was ignored.

**3.4 Version mismatch between banner and package.json**
`cli.ts:22` shows `v0.1.0` in the banner but `package.json` says `"version": "1.0.0"`. These are inconsistent.

### P1 - Should Fix

**3.5 `formatCategory` function duplicated in cli.ts and privacy-policy.ts**
Two different implementations of category-to-label mapping. The CLI version (cli.ts:46-74) includes emoji icons; the privacy-policy version (privacy-policy.ts:120-135) is text-only. They should share a single source of truth for category labels.

**3.6 MCP server uses string concatenation for paths (mcp/server.ts:107)**
`` `${projectPath}/${outputDir}` `` instead of `path.join(projectPath, outputDir)`. Breaks on Windows and with trailing slashes.

**3.7 MCP server does not pass config to generateDocuments (mcp/server.ts:106)**
`generateDocuments(result)` is called without loading config, so MCP-generated documents always use placeholder values (`[Your Company Name]`). The CLI correctly loads config; the MCP server doesn't.

**3.8 `getExpectedFilename` in MCP server is missing DPA mapping (mcp/server.ts:238-245)**
The `check_compliance` tool maps doc names to filenames but doesn't include "Data Processing Agreement" -> "DATA_PROCESSING_AGREEMENT.md". Falls back to the dynamic mapping which happens to work, but it's inconsistent with the explicit approach for the other 4 docs.

**3.9 No input validation on projectPath in CLI (cli.ts:108)**
If the user passes a non-existent directory, `scan()` will try to read files and silently return empty results. No error message like "Directory does not exist."

### P2 - Nice to Fix

**3.10 Hardcoded ANSI color codes instead of using a library or respecting NO_COLOR (cli.ts:11-17)**
The CLI doesn't respect the `NO_COLOR` environment variable (de facto standard). Piping output to a file includes ANSI escape sequences.

**3.11 Watch mode creates recursive watchers without depth limit (cli.ts:321-348)**
On a large project, this could create thousands of file watchers and consume significant resources. `fs.watch` is also known to be unreliable on some platforms.

**3.12 No programmatic access to individual generators**
The public API (index.ts) exports `generateDocuments` but not individual generators like `generatePrivacyPolicy`. Users who only want one document can't import it directly (without reaching into internal paths).

## 4. Test Quality Issues

### P0 - Must Fix

**4.1 No tests for edge cases in env scanner matching logic**
The most dangerous bug (1.1 - loose env matching) has zero test coverage. No test for short env var names, no test for substring false positives. The existing tests only verify happy paths.

**4.2 No tests for the MCP server**
Zero test coverage for the MCP server. Three tools with error handling, path construction, and config loading -- none tested.

**4.3 No tests for the DPA generator**
`generateDPA` has no dedicated test. The orchestrator test (generator.test.ts:384-395) verifies it appears in the list of 5 documents, but no test checks DPA content, context usage, or edge cases.

### P1 - Should Fix

**4.4 Generator tests only check for substring presence, not document structure**
Every generator test uses `assert.ok(result.includes("..."))`. None verify: correct markdown structure, valid table formatting, correct section ordering, or absence of malformed content. A generator could produce broken markdown and all tests would pass.

**4.5 Dependency scanner test for scoped packages tests wrong behavior (dependencies.test.ts:76-83)**
Test "detects scoped packages" asserts `names.includes("@sentry/node")` when the actual dependency is `@sentry/nextjs`. The test is documenting and locking in the wrong behavior from issue 1.2.

**4.6 Integration tests don't verify document generation end-to-end**
`integration.test.ts` tests scanning but never tests `generateDocuments` or `writeDocuments`. No test creates a temp project, runs the full pipeline, and verifies the generated files exist and contain correct content.

**4.7 No negative tests for import scanner**
No test that verifies the import scanner does NOT match: comments containing import strings, strings inside template literals, or variable names that happen to match service names (e.g., `const stripe = "my custom stripe"`).

### P2 - Nice to Fix

**4.8 Test file count is only ~85 test cases, not 100 as mentioned in roadmap**
The roadmap says "ensure all 100 tests actually test meaningful behavior" but there appear to be fewer than 100 tests across all test files. Should count and reconcile.

**4.9 No performance tests**
No test measures scan time on a large project. Could regress without notice.

## 5. CLI UX Issues

### P1 - Should Fix

**5.1 No error message for invalid command (cli.ts:147-149)**
`codepliant foo` prints "Unknown command: foo" in red, then the full usage text. No suggestion like "Did you mean 'go'?"

**5.2 `codepliant scan` with --json outputs nothing on empty project**
When no services are detected with `--json`, it outputs `{"services":[],...}` which is correct, but without `--json` it prints "No services detected" and exits. Inconsistent -- the non-JSON path doesn't show the full scan result structure.

**5.3 `codepliant go` generates ToS even with zero services (generator/index.ts:49-54)**
If no services are detected, a Terms of Service is still generated. This is technically correct (ToS is always recommended) but confusing -- user sees "No services detected" followed by "1 document generated."

**5.4 Output directory is resolved relative to project path, not CWD (cli.ts:109)**
`const absOutputDir = path.resolve(absProjectPath, outputDir)` means `--output ./docs` creates `docs/` inside the scanned project, not relative to where the user ran the command. Could be confusing when scanning a different directory.

**5.5 Watch mode doesn't watch for new directories (cli.ts:321-348)**
The watcher recursively adds watchers for existing directories at startup but doesn't add watchers for newly created directories. If a user creates `src/newfeature/` and adds code there, watch mode won't detect it.

### P2 - Nice to Fix

**5.6 No `--version` flag**
`codepliant --version` doesn't work. Common CLI expectation.

**5.7 No `--dry-run` option for `go` command**
Users can't preview what would be generated without actually writing files.

**5.8 Init command doesn't validate email format**
`codepliant init` accepts any string for contact email, including empty or clearly invalid values.

## 6. MCP Server Issues

### P1 - Should Fix

**6.1 Config not loaded for document generation (see 3.7)**
Duplicate of code quality issue 3.7. Documents generated via MCP always use placeholder values.

**6.2 No tool for scanning without generating**
The MCP server has `scan_project` (scan only) and `generate_compliance_docs` (scan + generate + write). But `generate_compliance_docs` always writes to disk. No way to preview documents without writing.

**6.3 `check_compliance` only checks `/legal/` directory (mcp/server.ts:175)**
Hardcoded to check `legal/` directory. If the user configured a different output directory via `.codepliantrc.json`, the compliance check will always report documents as missing.

**6.4 No tool description includes parameter constraints**
Tool descriptions don't mention that `projectPath` must be an existing directory, or what happens on error.

### P2 - Nice to Fix

**6.5 MCP server doesn't expose scan results as MCP resources**
The scan result could be exposed as a resource (per the v7.0 roadmap) but currently isn't. Not blocking for v1.0 but worth noting.
