#### BUG-2 RESIDUAL: HIGH - Env var `v.includes(envPattern)` still causes false positives

**Severity:** High
**Repro:**
```bash
echo 'AUTH_SECRET_VALUE=xyz' > /tmp/test/.env.example
codepliant scan /tmp/test --json
```
**Result:** `AUTH_SECRET_VALUE` matches `@auth/core` because the check `v.includes(envPattern)` is true (the user's var name `AUTH_SECRET_VALUE` includes `AUTH_SECRET`). This is the remaining direction of the `includes()` match.
**Assessment:** PM has decided this direction is acceptable (a user var containing the pattern is a reasonable signal). The remaining false positive rate is low. **Accepted per PM decision.**

#### BUG-3: HIGH - DATABASE_URL triggers both Prisma AND Drizzle detection

**Severity:** High
**Status:** Still present.
**Repro:**
```bash
echo 'DATABASE_URL=postgresql://localhost/mydb' > /tmp/test/.env.example
codepliant scan /tmp/test --json
```
**Result:** Both `prisma` and `drizzle` detected from a single env var. Also would match `SQLAlchemy` and `psycopg2` from Python signatures.
**Root cause:** `DATABASE_URL` appears in env patterns for prisma, drizzle, mongoose (Node), SQLAlchemy, prisma, psycopg2 (Python). It's too generic to attribute to a specific service.
**Recommendation:** Remove `DATABASE_URL` from env patterns for specific ORMs, or create a generic "Database" detection for shared vars, or require corroborating dependency/import evidence.

#### BUG-4: HIGH - Import scanner matches string literals (36 false positives on self-scan)

**Severity:** High
**Status:** Still present.
**Repro:**
```bash
codepliant scan /path/to/codepliant --json
# Returns 36 services, all false positives from types.ts, ai-disclosure.ts, etc.
```
**Root cause:** The JS import regex `(?:import|require|from)\s*[("'\`].*${pattern}.*[)"'\`]` is too broad. It matches any line where `from` appears followed by a quoted string containing the pattern. In `types.ts`, the SERVICE_SIGNATURES object has lines like `"@anthropic-ai/sdk": {` which are object keys, not imports.

Additionally, the `usageRegex` (`new Anthropic`) matches string literals like `"Anthropic"` in provider-name mappings.

**Impact:** Any project that has configuration files listing service names as strings will get false positives.
**Recommendation:** Require the regex to match at line start (or after whitespace only), and ensure `from` is followed by a proper import path syntax.

#### BUG-5: MEDIUM - CLI crashes on file path instead of directory

**Severity:** Medium
**Status:** Still present.
**Repro:**
```bash
codepliant go /path/to/file.txt
```
**Result:** `ENOTDIR: not a directory, mkdir '/path/to/file.txt/legal'`
**Fix:** Add validation in `cli.ts` after resolving `absProjectPath`:
```typescript
if (!fs.existsSync(absProjectPath) || !fs.statSync(absProjectPath).isDirectory()) {
  console.error(`Error: ${absProjectPath} is not a directory`);
  process.exit(1);
}
```

#### BUG-8: MEDIUM - Scoped package false matching in dependency scanner

**Severity:** Medium
**Status:** Still present (flagged by PM as item 1.2).
**Repro:**
```bash
echo '{"dependencies":{"@sentry/nextjs":"^8.0.0"}}' > /tmp/test/package.json
codepliant scan /tmp/test --json
```
**Result:** Detects `@sentry/node` with evidence `@sentry/nextjs@^8.0.0`. The dependency scanner matches `@sentry/nextjs` against the `@sentry/node` signature because of the logic: `dep.startsWith(sigName.split("/")[0] + "/")` which checks if the dep starts with `@sentry/`.
**Impact:** Wrong service name displayed. Data collected and category are still correct (both are monitoring/Sentry), but the detected name is misleading.
**Fix:** The matching logic should check exact match or have explicit aliases.

## EDGE CASES THAT NEED HANDLING

1. **Path validation** -- CLI does not validate project path is an existing directory before scanning.
2. **Shared env vars** -- `DATABASE_URL` is used by many services. Env-only detection for these vars produces false positives.
3. **Import regex precision** -- Import scanner regex matches string content in object literals, not just actual imports.
4. **Read-only filesystem** -- `writeDocuments` doesn't handle `EACCES` errors gracefully.
5. **Unicode/binary source files** -- Scanner reads all files with `utf-8` encoding. Binary files in source dirs (with `.ts`/`.js` extension) could cause issues. Builder added binary extension blocklist but `.ts`/`.js` files with binary content are not handled.
6. **Concurrent watch mode changes** -- Watch mode debounces at 500ms but doesn't queue -- rapid changes could be missed.
