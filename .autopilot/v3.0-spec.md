## Background

The v1.0 legal audit (`.autopilot/v1.0-legal-audit.md`) found the AI Disclosure generator missing critical EU AI Act requirements: Art. 50(2) synthetic content marking, Art. 50(3) emotion recognition, Art. 50(4) deepfake disclosure, and Art. 50(5) first-interaction accessibility. The current template in `src/generator/ai-disclosure.ts` is generic boilerplate — it lists risk categories without classifying the user's app, and provides no actionable guidance.

EU AI Act full enforcement date: **2 August 2026** (less than 5 months away). Penalties up to EUR 35M or 7% global annual turnover.

## 2. AI Risk Classification Logic

**File:** `src/generator/ai-disclosure.ts` (new function: `classifyAIRisk`)

The generator must auto-classify the app's AI risk level based on detected services and patterns. If `aiRiskLevel` is set in config, use that instead (override).

### Classification Rules

#### High Risk
Classify as **high** if ANY of these conditions are true:
- Config `aiRiskLevel` is explicitly set to `"high"`
- Detected AI services are combined with services in sensitive domains. Since we cannot yet detect the domain from code alone, high risk is only assigned via config override or future domain detection (v5.0+).

For now, high risk requires manual override. The generated document should note: *"Automatic high-risk detection is limited. If your application uses AI in employment decisions, educational assessment, credit scoring, healthcare diagnosis, or law enforcement, set `aiRiskLevel: 'high'` in your codepliant config."*

#### Limited Risk (default when AI is detected)
Classify as **limited** if:
- Any AI service is detected (openai, @anthropic-ai/sdk, @google/generative-ai, replicate, together-ai, cohere, langchain)
- AND `aiRiskLevel` is not set to something else

This is the correct default because most SaaS apps using AI have user-facing chatbots, content generation, or recommendation features — all of which trigger Art. 50 transparency obligations.

#### Minimal Risk
Classify as **minimal** if:
- Only vector database / embedding services are detected (@pinecone-database/pinecone) with NO generative AI service
- OR config `aiRiskLevel` is explicitly set to `"minimal"`

Minimal risk applies when AI is used purely for internal analytics, search optimization, or backend processing with no user-facing AI interaction.

### Classification Function Signature

```typescript
type AIRiskLevel = "minimal" | "limited" | "high";

interface AIRiskClassification {
  level: AIRiskLevel;
  reasoning: string;
  detectedServices: string[];
  isOverride: boolean;  // true if set via config
}

function classifyAIRisk(
  aiServices: DetectedService[],
  configOverride?: AIRiskLevel
): AIRiskClassification
```

### Reasoning Text by Level

- **Minimal:** `"Only embedding/vector services detected (no generative AI). AI is used for internal processing without direct user interaction."`
- **Limited:** `"Generative AI services detected ([list]). Users may interact with AI-generated content, triggering EU AI Act Article 50 transparency obligations."`
- **High:** `"Manually classified as high risk. Applications using AI for [employment/education/healthcare/credit/law enforcement] decisions must comply with EU AI Act Chapter III requirements."`
- **Override (any level):** Prepend `"Risk level manually set via configuration. "` to the reasoning.

## 3. AI Disclosure Rewrite

**File:** `src/generator/ai-disclosure.ts`

Complete rewrite. The document must contain these sections in this order. Use auto-incrementing section numbers (same pattern as v2.0 privacy policy rewrite).

### Section 1: Header and Preamble

```
# AI Disclosure Statement

**Last updated:** ${date}

**Project:** ${scan.projectName}

---

> This disclosure is provided in compliance with the EU AI Act (Regulation 2024/1689). Transparency obligations under Article 50 take full effect on 2 August 2026. Penalties for non-compliance reach up to EUR 35 million or 7% of global annual turnover.
```

### Section 2: AI Systems Inventory (auto-generated)

```
## ${n}. AI Systems in Use

This application integrates the following AI systems:

| AI Service | Provider | Type | Data Processed |
|-----------|----------|------|---------------|
| ${service.name} | ${provider} | ${serviceType} | ${dataCollected} |
```

The `Type` column is new. Map services to types:
- openai, @anthropic-ai/sdk, @google/generative-ai, cohere, together-ai → "Large Language Model"
- replicate → "Multi-modal AI"
- langchain → "AI Orchestration Framework"
- @pinecone-database/pinecone → "Vector Database"

### Section 3: Risk Classification

```
## ${n}. Risk Classification

Under the EU AI Act, this application's AI usage is classified as:

**${level} Risk**

**Reasoning:** ${classification.reasoning}

### Risk Level Definitions

| Level | Description | Obligations |
|-------|------------|-------------|
| Minimal | Internal analytics, search optimization, spam filtering | No specific obligations |
| Limited | Chatbots, content generation, recommendations | Transparency and disclosure (Article 50) |
| High | Employment, education, healthcare, credit scoring, law enforcement | Full conformity assessment, registration, monitoring (Chapter III) |
```

If level is "high", add:

```
> **Important:** High-risk AI systems must undergo conformity assessment, be registered in the EU database, and maintain ongoing risk management and monitoring. Consult legal counsel for full Chapter III compliance.
```

If level is "limited", add:

```
> **Your obligations:** As a limited-risk AI system, you must comply with the transparency requirements detailed in the following sections. These are mandatory, not optional.
```

### Section 4: Transparency Obligations

```
## ${n}. Transparency Obligations

In accordance with Article 50 of the EU AI Act:
```

#### 4.1: User Interaction Disclosure (Art. 50(1)) — always included

```
### ${n}.1 User Interaction Disclosure (Article 50(1))

Users must be informed when they are interacting with an AI system, unless this is obvious from the circumstances. This application uses AI in the following user-facing features:

- [Describe AI-powered features — e.g., chat assistant, content suggestions, automated responses]

> **Action required:** Ensure a visible notice is displayed to users before or at the point of AI interaction. Example: "This response was generated by AI" or an AI indicator badge.
```

If `aiUsageDescription` is configured, use that text instead of the placeholder list.

#### 4.2: Synthetic Content Marking (Art. 50(2)) — always included when generative AI detected

```
### ${n}.2 Synthetic Content Marking (Article 50(2))

AI-generated text, images, audio, or video must be marked as artificially generated in a machine-readable format. This applies to all output from:

${list of generative AI services detected}

**Requirements:**
- Outputs must be marked using machine-readable metadata (e.g., C2PA standard, watermarking, embedded flags)
- Marking must be robust, interoperable, and reliably detectable
- The EU Code of Practice on AI-generated content labelling (final version expected June 2026) will establish specific technical standards

> **Action required:** Implement a technical mechanism to mark AI-generated outputs as synthetic. Options include:
> - [C2PA metadata](https://c2pa.org/) for images and video
> - Watermarking for text content
> - Response headers or metadata flags for API outputs
> - Visible labels combined with machine-readable markers
```

Only include this section if at least one generative AI service is detected (openai, @anthropic-ai/sdk, @google/generative-ai, cohere, together-ai, replicate). Do NOT include for vector-database-only setups.

#### 4.3: Emotion Recognition Disclosure (Art. 50(3)) — conditional

```
### ${n}.3 Emotion Recognition Disclosure (Article 50(3))

If this application uses AI-based emotion recognition or biometric categorization systems, the following applies:

- Individuals exposed to such systems must be informed of their operation
- The information must be provided prior to or at the point of exposure
- Personal data processed through emotion recognition is subject to GDPR

> **Does this apply to you?** This section is included because AI services were detected. If your application analyzes facial expressions, voice tone, body language, or other biometric signals to infer emotions, this disclosure is mandatory. If not, you may remove this section.
```

Include this section ONLY if risk level is "limited" or "high". It is informational/conditional — the scanner cannot yet detect emotion recognition usage specifically.

#### 4.4: Deep Fake Disclosure (Art. 50(4)) — conditional

```
### ${n}.4 Synthetic Media Disclosure (Article 50(4))

If this application generates or manipulates image, audio, or video content that resembles existing persons, places, or events ("deep fakes"), the following applies:

- Such content must be clearly labeled as artificially generated or manipulated
- The label must be visible at the point of publication or distribution
- Exceptions exist for artistic, satirical, or fictional works, provided they do not mislead

> **Does this apply to you?** If your application generates realistic images of people, voice clones, or manipulated video, this disclosure is mandatory. If your AI usage is limited to text generation, you may remove this section.
```

Include this section ONLY if `replicate` or `@google/generative-ai` is detected (multi-modal services), OR if risk level is "high". Text-only LLM services (openai, anthropic, cohere) do NOT trigger this section by default.

#### 4.5: First-Interaction Disclosure Placement (Art. 50(5)) — always included

```
### ${n}.5 Disclosure Placement and Accessibility (Article 50(5))

Under Article 50(5), all transparency information must be:

- Provided at the **point of first interaction** with the AI system
- Presented in a **clear and distinguishable** manner
- Compliant with **accessibility requirements** under Directive (EU) 2019/882

**Recommended implementation:**

1. **In-app notification:** Display a notice before the user's first interaction with any AI feature (e.g., modal, banner, or inline notice)
2. **AI indicator:** Show a persistent AI badge or icon on AI-generated content (e.g., "AI" pill, sparkle icon)
3. **Link to full disclosure:** Link from the in-app notice to this full AI Disclosure document
4. **Accessibility:** Ensure notices are screen-reader compatible, use sufficient color contrast, and are available in the user's language
5. **Settings page:** Provide an AI settings or preferences page where users can review AI features and opt out where feasible
```

## 4. AI Act Compliance Checklist — New Document Type

**New file:** `src/generator/ai-compliance-checklist.ts`

This is a new document type: a checklist that users can follow to verify their EU AI Act compliance. It is generated alongside the AI Disclosure when AI services are detected.

### Generator Function Signature

```typescript
export function generateAIComplianceChecklist(
  scan: ScanResult,
  ctx?: GeneratorContext
): string | null
```

Returns `null` if no AI services detected (same pattern as AI Disclosure).

### Checklist Content

The checklist is dynamic based on risk classification. Use the same `classifyAIRisk` function.

```
# EU AI Act Compliance Checklist

**Project:** ${scan.projectName}
**Generated:** ${date}
**Risk Classification:** ${level} Risk
**Enforcement Date:** 2 August 2026

---

> Use this checklist to track your EU AI Act compliance progress. Items marked [REQUIRED] are legal obligations. Items marked [RECOMMENDED] are best practices.

## General Transparency (All Risk Levels)

- [ ] [REQUIRED] Users are informed when interacting with an AI system (Art. 50(1))
- [ ] [REQUIRED] AI disclosure document is published and accessible (Art. 50(5))
- [ ] [REQUIRED] Disclosure meets accessibility requirements (Directive 2019/882)
- [ ] [RECOMMENDED] AI indicator badge/icon displayed on AI-generated content
- [ ] [RECOMMENDED] AI features described in Terms of Service and Privacy Policy
```

If generative AI detected (limited or high risk), add:

```
## Content Marking (Generative AI)

- [ ] [REQUIRED] AI-generated text is identifiable as synthetic (Art. 50(2))
- [ ] [REQUIRED] AI-generated images/audio/video include machine-readable markers (Art. 50(2))
- [ ] [REQUIRED] Marking method is interoperable with detection tools
- [ ] [RECOMMENDED] Using C2PA or equivalent standard for media content
- [ ] [RECOMMENDED] Response metadata includes AI-generation flag for API outputs
```

If multi-modal AI detected (replicate, @google/generative-ai), add:

```
## Synthetic Media (Deep Fakes)

- [ ] [REQUIRED] Synthetic media resembling real persons/events is labeled (Art. 50(4))
- [ ] [REQUIRED] Labels are visible at point of publication
- [ ] [RECOMMENDED] Watermarking applied to generated images and video
```

Always include for limited and high risk:

```
## Emotion Recognition Check

- [ ] [REQUIRED] Confirm whether your application performs emotion recognition or biometric categorization
- [ ] [REQUIRED] If yes: inform exposed individuals prior to system operation (Art. 50(3))
- [ ] [REQUIRED] If yes: ensure GDPR compliance for biometric data processing
- [ ] If no: document that emotion recognition is not used (for audit trail)
```

If high risk:

```
## High-Risk AI Obligations (Chapter III)

- [ ] [REQUIRED] Risk management system established and maintained (Art. 9)
- [ ] [REQUIRED] Data governance measures implemented (Art. 10)
- [ ] [REQUIRED] Technical documentation prepared (Art. 11)
- [ ] [REQUIRED] Record-keeping / logging enabled (Art. 12)
- [ ] [REQUIRED] Transparency information provided to deployers (Art. 13)
- [ ] [REQUIRED] Human oversight measures in place (Art. 14)
- [ ] [REQUIRED] Accuracy, robustness, and cybersecurity ensured (Art. 15)
- [ ] [REQUIRED] Conformity assessment completed (Art. 43)
- [ ] [REQUIRED] AI system registered in EU database (Art. 49)
- [ ] [REQUIRED] Post-market monitoring system operational (Art. 72)
- [ ] [RECOMMENDED] Appointed an AI compliance officer or team
- [ ] [RECOMMENDED] Conducted fundamental rights impact assessment (Art. 27)
```

Always include:

```
## Data Processing & Privacy

- [ ] [REQUIRED] Privacy Policy references AI data processing
- [ ] [REQUIRED] Data Processing Agreements in place with AI providers
- [ ] [REQUIRED] International data transfer safeguards documented (SCCs, DPF)
- [ ] [RECOMMENDED] Data minimization applied to AI inputs
- [ ] [RECOMMENDED] User data not used for model training (or disclosed if it is)

## Detected AI Providers — Provider-Specific Actions

${for each AI service:}

- [ ] Review [${provider} Privacy Policy](${privacyUrl})
- [ ] Review [${provider} Terms / Data Processing](${termsUrl})
- [ ] Confirm data retention settings (disable training on your data if available)

*Generated by [Codepliant](https://github.com/codepliant/codepliant). This checklist is a compliance aid, not legal advice. Consult legal counsel for your specific obligations under the EU AI Act.*
```

## 6. Implementation Notes for Builder

1. **Do config first.** The disclosure and checklist both depend on `aiRiskLevel` and `aiUsageDescription` being available in `GeneratorContext`.

2. **Extract `classifyAIRisk` as a shared function.** Both the disclosure and checklist need it. Put it in `ai-disclosure.ts` and export it, or create a small `src/generator/ai-risk.ts` utility. Builder's call.

3. **Use auto-incrementing section numbers.** Same `let n = 1` pattern used in the v2.0 privacy policy rewrite. Do not hardcode section numbers.

4. **Conditional sections.** Sections 4.3 (emotion recognition) and 4.4 (deep fakes) are conditional. Use the detection rules specified in Section 3 of this spec. When a conditional section is excluded, section numbering must still be contiguous.

5. **The `aiUsageDescription` config replaces the placeholder in Section 4.1.** If not set, keep the placeholder with the "Action required" callout.

6. **Provider URL mapping.** The current `getProviderPolicyUrl` function returns a single URL. Replace it with the `PROVIDER_URLS` map that includes both `privacy` and `terms` URLs. Add `langchain` which is currently missing from the provider maps.

7. **Do NOT touch other generators** (privacy-policy, terms-of-service, cookie-policy, dpa) in this version.

8. **Test with these scenarios:**
   - OpenAI only → limited risk, all transparency sections, no deep fake section
   - Pinecone only → minimal risk, reduced sections
   - OpenAI + Replicate → limited risk, includes deep fake section
   - Config override to "high" → high risk, full checklist with Chapter III items
   - No AI services → both generators return null
