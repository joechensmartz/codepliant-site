# v5.0 Spec — Smart Scanning + Output Formats

**Version:** 5.0
**Date:** 2026-03-15
**Status:** READY FOR BUILD

---

## Overview

v5.0 has two major areas:

1. **Smart Scanning** — go beyond package detection to understand what user data the app actually stores, what endpoints collect it, and what tracking scripts are embedded in the frontend.
2. **Output Formats & User Experience** — users don't just want Markdown files in a `legal/` folder. They want HTML they can embed on their website, PDF they can send to lawyers/investors, and JSON they can consume via API. The boss specifically requested this: "not just code — also PDF and others, whatever users actually care about."

Both areas pull up from roadmap v5.0 and v6.0 respectively, merged here because output formats are the higher user-impact item.

## Area 1: Smart Scanning

### 1.1 Prisma Schema Scanning (HIGH PRIORITY)

**File:** `src/scanner/prisma.ts` (new)

Scan `prisma/schema.prisma` (and `schema.prisma` in project root) for user data fields. This tells us *what* data the app stores, not just *which services* it uses.

#### Detection targets

| Field pattern | Data category | Example matches |
|---|---|---|
| `email` | Personal Identity | `email String`, `userEmail String` |
| `phone`, `mobile`, `tel` | Contact Information | `phone String?`, `mobileNumber String` |
| `address`, `street`, `city`, `zip`, `postal`, `country`, `state` | Location / Address | `billingAddress String` |
| `name`, `firstName`, `lastName`, `fullName` | Personal Identity | `name String` |
| `password`, `hash`, `salt` | Credentials | `passwordHash String` |
| `dob`, `dateOfBirth`, `birthDate`, `age` | Sensitive Personal Data | `dateOfBirth DateTime?` |
| `ssn`, `socialSecurity`, `taxId`, `nationalId` | Government ID | `ssnLast4 String?` |
| `ip`, `ipAddress` | Technical / Network | `lastLoginIp String?` |
| `avatar`, `profilePic`, `photo` | Biometric-adjacent | `avatarUrl String?` |
| `gender`, `sex`, `ethnicity`, `race`, `religion` | Special Category (GDPR Art. 9) | `gender String?` |
| `creditCard`, `cardNumber`, `cvv`, `iban`, `bankAccount` | Financial | `cardLast4 String?` |
| `cookie`, `session`, `token`, `refreshToken` | Session / Auth | `sessionToken String` |
| `lat`, `lng`, `latitude`, `longitude`, `coordinates`, `location` | Geolocation | `lat Float?` |

#### Implementation

```typescript
// src/scanner/prisma.ts

export interface PrismaField {
  modelName: string;       // e.g., "User"
  fieldName: string;       // e.g., "email"
  fieldType: string;       // e.g., "String"
  dataCategory: string;    // e.g., "Personal Identity"
  isOptional: boolean;
}

export interface PrismaSchemaResult {
  schemaPath: string;
  models: string[];
  userDataFields: PrismaField[];
  hasUserModel: boolean;     // true if a model named "User", "Account", "Customer", or "Profile" exists
}

export function scanPrismaSchema(projectPath: string): PrismaSchemaResult | null
```

**Parsing approach:** Line-by-line regex. No need for a full Prisma parser.

1. Find `model X {` lines — extract model name
2. Inside model blocks, match field lines: `fieldName FieldType` (with optional `?`, `@` attributes)
3. Match field names against the pattern table (case-insensitive substring match)
4. Look specifically for models named `User`, `Account`, `Customer`, `Profile`, `Member` — these are the primary user data models

**File search order:**
1. `prisma/schema.prisma`
2. `schema.prisma` (project root)
3. `src/prisma/schema.prisma`
4. `db/schema.prisma`

#### Integration with ScanResult

Add a new optional field to `ScanResult`:

```typescript
export interface ScanResult {
  // ... existing fields
  userDataFields?: PrismaField[];   // from Prisma schema scanning
  dataFlowSummary?: DataFlowSummary; // from data flow mapping (Section 1.4)
}
```

The generator can use `userDataFields` to produce much more specific privacy policy language. Instead of "user data as defined in schema," we can say "email addresses, phone numbers, billing addresses, and date of birth."

### 1.2 API Route Scanning

**File:** `src/scanner/api-routes.ts` (new)

Scan API route files for data intake endpoints — POST/PUT/PATCH routes that accept user data in the request body.

#### Framework detection

| Framework | Route file patterns |
|---|---|
| Next.js App Router | `app/**/route.ts`, `app/**/route.js` |
| Next.js Pages Router | `pages/api/**/*.ts`, `pages/api/**/*.js` |
| Express | `*.ts`/`*.js` files containing `app.post(`, `router.post(` |
| Fastify | `*.ts`/`*.js` files containing `fastify.post(` |
| Hono | `*.ts`/`*.js` files containing `app.post(`, `.post(` with Hono import |

#### Detection targets

Look for request body field access patterns:

```
req.body.email          → Personal Identity
req.body.password       → Credentials
request.json().phone    → Contact Information
body.address            → Location
formData.get('file')    → User Uploads
```

Also detect:
- `z.object({` (Zod schema validation) — parse field names from the schema
- `@Body()` decorators (NestJS)
- `request.formData()` (file uploads)

#### Output

```typescript
export interface ApiEndpoint {
  file: string;
  method: "POST" | "PUT" | "PATCH";
  path: string;           // e.g., "/api/users", "/api/checkout"
  bodyFields: string[];   // e.g., ["email", "password", "name"]
  dataCategories: string[];
}

export function scanApiRoutes(projectPath: string): ApiEndpoint[]
```

### 1.3 HTML/JSX Inline Tracking Script Detection

**File:** `src/scanner/tracking.ts` (new)

Scan HTML, JSX, and TSX files for hardcoded tracking scripts that the dependency/import scanners miss. Many developers add tracking via `<script>` tags or `dangerouslySetInnerHTML` rather than npm packages.

#### Detection targets

| Pattern | Service | Data collected |
|---|---|---|
| `gtag.js`, `googletagmanager.com`, `UA-`, `G-`, `GTM-` | Google Analytics / GTM | Page views, user behavior, device info, IP |
| `fbq(`, `connect.facebook.net`, `facebook.com/tr` | Meta Pixel | Page views, conversions, user behavior, hashed PII |
| `snap.licdn.com`, `linkedin.com/insight` | LinkedIn Insight | Page views, conversions, professional profile |
| `static.hotjar.com`, `hj(` | Hotjar | Session recordings, heatmaps, form interactions |
| `plausible.io` | Plausible | Page views (privacy-friendly) |
| `sc.lkqd.net`, `ads.twitter.com`, `t.co/i/adsct` | Twitter/X Pixel | Conversions, page views |
| `snap.licdn.com` | Snapchat Pixel | Conversions, page views |
| `tiktok.com/i18n/pixel` | TikTok Pixel | Conversions, page views, device info |
| `clarity.ms` | Microsoft Clarity | Session recordings, heatmaps |
| `mc.yandex.ru/metrika` | Yandex Metrica | Page views, user behavior |

#### Implementation

Scan files matching: `**/*.html`, `**/*.htm`, `**/*.jsx`, `**/*.tsx`, `src/**/layout.{tsx,jsx,ts,js}`, `app/**/layout.{tsx,jsx,ts,js}`, `pages/_app.{tsx,jsx,ts,js}`, `pages/_document.{tsx,jsx,ts,js}`.

Use regex to find `<script` tags and `dangerouslySetInnerHTML` content. Match against the pattern table.

```typescript
export interface InlineTracker {
  file: string;
  service: string;
  trackingId?: string;     // extracted GA ID, Pixel ID, etc.
  dataCollected: string[];
}

export function scanTrackingScripts(projectPath: string): InlineTracker[]
```

**Convert to DetectedService:** Each detected tracker becomes a `DetectedService` with `evidence.type = "code_pattern"`, merged into the main scan results.

### 1.4 Data Flow Mapping

**File:** `src/scanner/data-flow.ts` (new)

Synthesize findings from all scanners into a data flow summary: collection → storage → sharing.

```typescript
export interface DataFlowSummary {
  collection: DataFlowEntry[];   // where data enters the system
  storage: DataFlowEntry[];      // where data is persisted
  sharing: DataFlowEntry[];      // where data leaves to third parties
}

export interface DataFlowEntry {
  dataType: string;              // e.g., "email addresses"
  source: string;                // e.g., "Registration form (/api/auth/register)"
  destination: string;           // e.g., "PostgreSQL via Prisma", "SendGrid"
  purpose: string;               // e.g., "Account creation", "Marketing emails"
}

export function buildDataFlowSummary(
  services: DetectedService[],
  userDataFields?: PrismaField[],
  apiEndpoints?: ApiEndpoint[],
  inlineTrackers?: InlineTracker[]
): DataFlowSummary
```

**Logic:**
- **Collection:** API endpoints that accept user data + auth services + payment forms
- **Storage:** Database services (Prisma, Drizzle, Mongoose) + Prisma schema fields
- **Sharing:** All third-party services (analytics, AI, email, payment, monitoring) — these send data to external servers

The generator uses this summary to produce a "Data Flow" section in the privacy policy, replacing the current generic "we collect data through X services" language.

### 1.5 Scanner Integration

Update `src/scanner/index.ts` to call the new scanners and merge results:


## Area 2: Output Formats & User Experience

### 2.1 Multiple Output Formats (HTML = HIGH PRIORITY)

Currently all output is Markdown. Add support for:

#### 2.1.1 HTML Output (HIGHEST PRIORITY)

**File:** `src/output/html.ts` (new)

Convert generated Markdown documents to styled HTML, ready to embed on a website.

```typescript
export function markdownToHtml(markdown: string, options?: HtmlOptions): string

export interface HtmlOptions {
  title?: string;
  customCss?: string;       // user-provided CSS to inject
  includeDefaultStyles?: boolean;  // default: true
  lastUpdated?: string;     // date string for footer
  companyName?: string;     // for page title / header
}
```

**Default styling:**
- Clean, readable typography (system font stack)
- Max-width container (720px), centered
- Responsive (works on mobile)
- Light background, dark text
- Styled headings, tables, lists
- "Last updated: [date]" footer
- Print-friendly (no background colors in print media)
- No external dependencies (all CSS inline in `<style>`)

**Markdown-to-HTML conversion:** Use a simple built-in converter (no external dependency). We only need to handle the Markdown subset we generate:
- Headings (`#` through `####`)
- Paragraphs
- Bullet lists and numbered lists
- Bold and italic
- Tables (pipe-delimited)
- Horizontal rules
- Links

Implement as a ~150-line function. Our generated Markdown is predictable and well-structured, so a full Markdown parser is unnecessary.

#### 2.1.2 PDF Output

**File:** `src/output/pdf.ts` (new)

Generate PDF from the HTML output. This is for legal review, investor due diligence, and record-keeping.

**Approach:** Since we have zero runtime dependencies, use one of:

**Option A (recommended): HTML-to-PDF via user's system.**
Provide a `--pdf` flag that uses Puppeteer (as a peer/optional dependency) or outputs HTML with a "print to PDF" instruction. Since we can't add runtime deps, the pragmatic approach is:

1. Generate the HTML file (from 2.1.1)
2. If `puppeteer` is available in the project or globally, use it to render PDF
3. Otherwise, output the HTML and log: "Open [file].html in a browser and print to PDF, or install puppeteer globally for automatic PDF generation"

**Option B: Minimal PDF generation.**
Generate a basic PDF directly using raw PDF format (no library). This is feasible but limited — no complex layout, just text with basic formatting. Good enough for legal documents which are text-heavy.

**Recommendation:** Start with Option A. PDF generation from HTML is reliable and produces good results. The HTML output (2.1.1) is the prerequisite — get that working first, PDF follows naturally.

```typescript
export async function htmlToPdf(html: string, outputPath: string): Promise<void>
```

#### 2.1.3 JSON Output

**File:** `src/output/json.ts` (new)

Structured JSON output for API consumption. Useful for:
- Displaying compliance docs in a SaaS dashboard
- Feeding into other tools
- Programmatic access to document sections

```typescript
export interface JsonDocument {
  documentType: string;      // e.g., "privacy-policy"
  title: string;

    jurisdiction: string[];

export interface JsonSection {
  id: string;              // e.g., "data-collection"

export function markdownToJson(markdown: string, documentType: string, metadata: Record<string, unknown>): JsonDocument
```

### 2.2 Hosted Compliance Page

## Output Pipeline Architecture

The current flow is: Scan → Generate (Markdown) → Write files.

The new flow adds an output transformation step:

```
Scan → Generate (Markdown) → Transform (format) → Write files

                             ├── markdown → *.md (current behavior)
                             ├── html     → *.html (styled)
                             ├── pdf      → *.pdf (via HTML)
                             ├── json     → *.json (structured)
                             └── all      → all of the above
```

**File:** `src/output/index.ts` (new)

```typescript
export interface OutputOptions {
  format: "markdown" | "html" | "pdf" | "json" | "all";
  generateHtmlPage: boolean;
  customCss?: string;
  companyName: string;
}

export async function transformAndWrite(
  documents: GeneratedDocument[],
  outputDir: string,
  options: OutputOptions
): Promise<string[]>   // returns list of written file paths
```

This replaces the direct call to `writeDocuments()` in the CLI and adds the format transformation layer.

## Notes for Builder

- **Zero runtime dependencies** remains a hard constraint. The HTML output, JSON output, Markdown-to-HTML converter, and compliance page must all be implemented without external packages. The Markdown subset we generate is small and predictable — a full parser is not needed.
- **PDF is the exception:** PDF generation may require Puppeteer as an optional/peer dependency. Implement it last and make it gracefully degrade (generate HTML + instructions if Puppeteer is not available).
- **Prisma parsing:** Do NOT use `@prisma/internals` or any Prisma library. Parse the schema file with regex. It's a simple format: `model Name { ... }` with `fieldName Type @attributes` lines.
- **File extensions for HTML output:** Use `.html` extension. For example, `PRIVACY_POLICY.html` alongside `PRIVACY_POLICY.md`.
- **The existing `writeDocuments()` function** in `src/generator/index.ts` should remain working for backward compatibility. The new output pipeline wraps it, not replaces it.
- **Do NOT modify existing generator files** (privacy-policy.ts, etc.) in Phase 1-2. The output format transformation works on the Markdown output they already produce. In Phase 3, the generators can be updated to leverage new scanner data (userDataFields, data flow).

## Notes for QA

- After HTML output is built, open generated `.html` files in a browser. Verify: headings render correctly, tables are styled, links work, page is responsive (resize window), print preview looks clean.
- After Prisma scanning is built, test with a real Prisma schema containing User, Post, Comment models. Verify: email, name, phone fields are detected; Post.title is NOT flagged as user data; optional fields are marked as optional.
- After JSON output is built, validate the JSON is valid (`JSON.parse()` succeeds) and the section structure matches the Markdown heading hierarchy.
- Test `--format all` generates .md + .html + .json for each document.
- Test that `--format markdown` (default) produces identical output to current behavior (no regression).
- Test compliance page: all documents appear in navigation, tab switching works, page renders on mobile viewport.
