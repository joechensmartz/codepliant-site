## 1. How Do Developers/Companies Use Compliance Documents?

### Placement Best Practices

Privacy policies and legal docs are placed in these locations:

1. **Website footer** — Universal standard. Link on every page. Required by GDPR (must be "readily available, easy to access, free of charge").
2. **Dedicated page** — `/privacy`, `/terms`, `/legal`. Most common for SaaS. Stripe uses `stripe.com/privacy`, Vercel uses `vercel.com/legal/privacy-policy`.
3. **Sign-up / checkout forms** — Inline link with checkbox for consent. Required for GDPR lawful basis of consent.
4. **Cookie banners** — Link to cookie policy from the consent banner.
5. **App store listings** — Apple App Store and Google Play both require a privacy policy URL.
6. **Email footers** — CAN-SPAM and GDPR require privacy policy link in marketing emails.

### How Startups Handle Legal Docs

- **Early stage:** Use generators (Termly, iubenda) or copy templates. Many use free generators and never update.
- **Growth stage:** Have a lawyer review generator output. Treat as "good enough" baseline.
- **Scale stage:** Custom-written by legal team. Hosted on their own site.
- **Common pattern:** Generate a document, paste it into a page on their site, forget about it until a compliance audit.

### Hosting Approaches

| Approach | Who Uses It | Pros | Cons |
|----------|------------|------|------|
| Self-hosted page on own site | Stripe, Vercel, Linear, most SaaS | Full control, professional look, SEO | Must update manually |
| Generator-hosted URL | Termly, iubenda free tier users | Auto-updates, zero setup | Branded, looks unprofessional, external domain |
| Embedded widget (iframe) | Small business sites, WordPress | Easy integration, auto-updates | Limited styling, iframe issues, third-party dependency |
| PDF download | Enterprise contracts, app stores | Offline access, formal feel | Hard to update, not web-accessible |
| GitHub/repo markdown | Open-source projects | Version-controlled, lives with code | Not user-facing, no styling |

**Key insight for Codepliant:** Developers who use our CLI tool are building apps. They want files they can drop into their project — either markdown for their repo, HTML for their site, or PDF for formal use. The markdown-only approach we have now covers the repo use case but misses the other two.

## 2. What Output Formats Do Competitors Offer?

### Termly

Termly offers three delivery methods:
1. **Hosted page (URL)** — Termly-hosted URL, always shows latest version. Auto-updates.
2. **Embed widget (code snippet)** — iframe that loads inline on your page. Auto-updates.
3. **HTML format** — Copy raw HTML, paste into your site. Most flexible for styling. Does NOT auto-update.
4. **No PDF export** — Users must print-to-PDF themselves.

Platform integrations: WordPress plugin, Shopify embed, Wix, Squarespace, GoDaddy.

### iubenda

iubenda offers four methods:
1. **Standard embedding** — Button/link in footer opens policy in a modal/lightbox.
2. **Direct text embedding** — Full policy text injected into your page with your own CSS. Can strip all iubenda styling.
3. **Direct link** — Opens policy on iubenda.com. Used for app stores.
4. **API access** — Programmatic retrieval of policy content.
5. **No native PDF** — Must print-to-PDF.

Notable: Free plan has iubenda branding and limited clause selection. Cannot add custom text on free plan.

### Other Generators

| Generator | Output Formats |
|-----------|---------------|
| TermsFeed | HTML download, hosted page, WordPress plugin |
| GetTerms.io | Hosted page with permanent link |
| WebsitePolicies | HTML, DOCX, Plain Text, Markdown, PDF |
| FreePP | Hosted page with free link |
| Enzuzo | Hosted page, embed widget |

**Key insight:** WebsitePolicies is the only generator offering markdown output. No generator offers a self-contained HTML file you can just drop into your project. Most rely on hosted/iframe approaches that create vendor lock-in.

### How Major Companies Host Their Policies

**Stripe** (`stripe.com/privacy`):
- Self-hosted dedicated page
- Clean, minimalist design — typography-driven, large heading hierarchy
- Monochromatic (black on white), sans-serif fonts
- Table of contents with anchor links for navigation
- Tabular data for US Consumer Privacy Notice
- Strategic hyperlinks to Privacy Center for supplementary details
- Professional, consistent formatting throughout

**Vercel** (`vercel.com/legal/privacy-policy`):
- Self-hosted under `/legal/` path
- Minimal design, integrated with site's design system
- Part of a legal docs hub (privacy, terms, DPA, SLA all under `/legal/`)

**Pattern:** Professional companies always self-host on their own domain with their own design. The hosted/iframe approach screams "we didn't invest in compliance."

## 3. PDF Generation in Node.js

### The Problem

We generate markdown files. Users want PDF and HTML too. But we're a zero-dependency CLI tool — we can't ship Puppeteer (150-400MB Chromium binary).

### Options Analysis

| Library | Headless Browser? | Install Size | Approach | Node.js? | Verdict |
|---------|------------------|-------------|----------|----------|---------|
| **Puppeteer** | YES (Chromium) | ~400MB | HTML → PDF via Chrome | Yes | TOO HEAVY — violates our zero-dependency philosophy |
| **md-to-pdf** | YES (Puppeteer) | ~400MB | MD → HTML → PDF via Chrome | Yes | Same problem — wraps Puppeteer |
| **markdown-pdf** | YES (PhantomJS) | Large | MD → HTML → PDF via PhantomJS | Yes | PhantomJS is DEPRECATED — do not use |
| **PDFKit** | NO | ~2MB | Programmatic drawing API | Yes | LIGHTWEIGHT — but no HTML/MD parsing. Must manually place every element. |
| **pdfmake** | NO | ~3MB | Declarative document definition | Yes | GOOD — JSON-like document structure, no browser needed. Tables, lists, styling built-in. |
| **jsPDF** | NO | ~2MB | Imperative API | Browser-first | NOT IDEAL — browser-focused, limited CSS support |
| **pdf-lib** | NO | ~1MB | Low-level PDF manipulation | Yes | LIGHTWEIGHT — TypeScript, fast, but very low-level |

### Recommended Approach for Codepliant

**Option A: `marked` + `pdfmake` (RECOMMENDED)**
1. Use `marked` (already a well-known MD parser) to parse our generated markdown into tokens
2. Transform tokens into pdfmake's declarative document definition (JSON-like structure)
3. pdfmake handles rendering — supports headings, paragraphs, tables, lists, bold/italic
4. No browser needed. ~3MB total. Runs anywhere Node runs.

**Option B: `marked` + `PDFKit`**
1. Parse markdown with `marked`
2. Walk the token tree and emit PDFKit drawing commands
3. More control but more code — must handle line wrapping, page breaks, tables manually

**Option C: Skip PDF, just do HTML**
1. Use `marked` to convert MD to HTML
2. Wrap in a minimal HTML template with inline CSS
3. Users can open in browser and print-to-PDF themselves
4. Zero new dependencies. Simplest approach.

**Recommendation:** Start with Option C (HTML output). It's zero dependencies and covers 80% of the use case. Add pdfmake later if users demand native PDF. The HTML file can be opened in any browser and printed to PDF, so it's effectively both formats.

### Key Constraint

Our `package.json` has zero runtime dependencies. Any PDF library would need to be either:
- A devDependency only (used at build time)
- Bundled into our dist
- Or we skip it entirely and generate HTML instead

## 4. HTML Compliance Pages

### What Makes a Professional Legal Page

Based on analysis of Stripe, Airbnb, OpenAI, and other top-tier privacy pages:

**Design principles:**
- **Typography-first** — Large, clear heading hierarchy. No decorative elements.
- **Monochromatic** — Black text on white background. Maybe one accent color for links.
- **Generous whitespace** — Substantial margins and padding. Legal text needs breathing room.
- **Sans-serif fonts** — System fonts or Google Fonts (Inter, Roboto, etc.) for legibility.
- **Table of contents** — Anchor links at the top for navigation in long documents.
- **Responsive** — Must work on mobile (regulators check on phones too).

**Minimal CSS approach for our generated HTML:**
```
- System font stack (no external fonts needed)
- max-width: 800px, centered
- Line-height: 1.6-1.8 for readability
- Clear heading hierarchy (h1 > h2 > h3)
- Simple table styling with borders
- Print-friendly media query
- ~50-80 lines of CSS total
```

**Professional examples worth emulating:**
- **Airbnb** — "No frills" approach. Sections with clear titles, bullet-point details.
- **OpenAI** — Simple language, clear instructions, not overly complex.
- **Stripe** — Table of contents, tabular data for dense regulatory info, consistent formatting.

### Recommended HTML Template Structure

```
<!DOCTYPE html>
<html>
<head>
  <meta charset="utf-8">
  <meta name="viewport" content="width=device-width, initial-scale=1">
  <title>{Document Title} — {Company Name}</title>
  <style>/* ~60 lines of minimal CSS */</style>
</head>
<body>
  <main>
    <h1>{Document Title}</h1>
    <p class="meta">Last updated: {date}</p>
    <nav class="toc">{table of contents}</nav>
    {converted markdown content}
  </main>
</body>
</html>
```

Single file. No external dependencies. No frameworks. Just semantic HTML + minimal inline CSS. Opens in any browser, prints cleanly.

## 5. What Do Users Complain About With Existing Generators?

### Common Complaints (aggregated from forums, reviews, and analysis)

**1. Generic boilerplate that doesn't match the actual business**
> "Free generators describe an imaginary version of your business rather than what actually happens."

This is the #1 complaint. Generators use questionnaires and produce one-size-fits-all text. The output doesn't reflect what the code actually does. **This is exactly what Codepliant solves** — we scan actual code.

**2. Missing mandatory legal disclosures**
- GDPR: Missing legal basis per purpose, retention periods, data subject rights, international transfer safeguards
- CCPA: Missing 12-month data categories, sale/sharing disclosure, consumer rights
- Free generators skip these because they're complex to template

**3. No automatic updates when laws change**
- Free generators produce static text. Laws change (DUAA 2025, new US state laws). Documents go stale.
- Paid generators auto-update hosted versions but NOT downloaded copies.
- **Codepliant advantage:** Re-run the scan whenever you want. Code changes → document changes.

**4. Missing third-party service disclosures**
- Generators don't know you use Google Analytics, Stripe, Sentry, etc.
- They produce vague "we may use third-party services" language.
- **Codepliant advantage:** We detect actual services from code and name them specifically.

**5. Ugly, branded output**
- Free tiers have generator branding (iubenda logo, Termly branding)
- iframe embeds look out of place on professional sites
- Hosted pages are on the generator's domain, not yours
- No control over styling on free plans

**6. Paywall for essential features**
- GDPR clauses often behind paywall
- Cookie policy integration requires paid plan
- Multi-jurisdiction support is premium-only
- PDF/HTML download may require paid plan

**7. No version control / audit trail**
- Changes happen silently on hosted versions
- No diff between versions
- No git-friendly format
- **Codepliant advantage:** Markdown files in your repo = full git history

**8. Questionnaire fatigue**
- 50-100 questions to fill out
- Developers don't know the answers to legal questions
- Must consult lawyers anyway to answer the questionnaire correctly
- **Codepliant advantage:** Zero questionnaire. Scan the code. Done.

### What Users Actually Want (Synthesized)

1. **Accurate to their actual stack** — not generic boilerplate
2. **Multiple output formats** — markdown for repo, HTML for site, PDF for formal use
3. **Professional appearance** — no branding, clean design
4. **Easy to update** — re-run when code changes
5. **No vendor lock-in** — files they own, not hosted on someone else's domain
6. **Covers actual regulations** — not just "privacy policy" but jurisdiction-specific sections
7. **Developer-friendly** — CLI, CI/CD integration, version control

## 6. Prisma Schema Scanning for Data Fields

### What a Typical Prisma Schema Looks Like

```prisma
generator client {
  provider = "prisma-client-generate"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

model User {
  id        Int      @id @default(autoincrement())
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
  email     String   @unique
  name      String?
  password  String
  role      Role     @default(USER)
  posts     Post[]
  profile   Profile?
  orders    Order[]
}

model Profile {
  id        Int     @id @default(autoincrement())
  bio       String?
  avatar    String?
  phone     String?
  address   String?
  city      String?
  country   String?
  zipCode   String?
  userId    Int     @unique
  user      User    @relation(fields: [userId], references: [id])
}

model Order {
  id        Int      @id @default(autoincrement())
  createdAt DateTime @default(now())
  amount    Float
  userId    Int
  user      User     @relation(fields: [userId], references: [id])
}
```

### Personal Data Field Detection Patterns

Fields that indicate personal data collection (for privacy policy generation):

| Category | Field Name Patterns | GDPR Category |
|----------|-------------------|---------------|
| **Identity** | `name`, `firstName`, `lastName`, `fullName`, `displayName`, `username` | Basic identity |
| **Contact** | `email`, `emailAddress`, `phone`, `phoneNumber`, `telephone`, `mobile` | Contact data |
| **Address** | `address`, `street`, `city`, `state`, `country`, `zipCode`, `postalCode`, `region` | Location data |
| **Authentication** | `password`, `passwordHash`, `hashedPassword`, `salt`, `token`, `refreshToken` | Security credentials |
| **Demographics** | `dateOfBirth`, `dob`, `birthDate`, `age`, `gender`, `sex` | Special category (age) |
| **Financial** | `creditCard`, `cardNumber`, `bankAccount`, `iban`, `billingAddress`, `amount`, `price` | Financial data |
| **Biometric** | `fingerprint`, `faceId`, `biometric`, `voicePrint` | Special category (biometric) |
| **Device/Network** | `ipAddress`, `ip`, `userAgent`, `deviceId`, `macAddress`, `deviceFingerprint` | Technical identifiers |
| **Social** | `socialSecurityNumber`, `ssn`, `taxId`, `nationalId`, `passportNumber`, `driverLicense` | Government IDs |
| **Tracking** | `cookieId`, `sessionId`, `trackingId`, `advertisingId`, `gclid`, `fbclid` | Online identifiers |
| **Health** | `healthData`, `medicalRecord`, `diagnosis`, `prescription`, `bloodType` | Special category (health) |
| **Behavioral** | `searchHistory`, `browsingHistory`, `purchaseHistory`, `preferences`, `interests` | Behavioral profiling |
| **Media** | `avatar`, `profilePicture`, `photo`, `image` | Visual identity |

### Prisma Schema Parsing Approach

Prisma schema files (`prisma/schema.prisma` or `*.prisma`) follow a predictable syntax:

```
model ModelName {
  fieldName  FieldType  @attributes
}
```

**Scanning strategy:**
1. Find `schema.prisma` files in the project
2. Parse `model` blocks with regex: `model\s+(\w+)\s*\{([^}]+)\}`
3. Extract field names from each model: `(\w+)\s+(String|Int|Float|Boolean|DateTime|...)`
4. Match field names against the personal data patterns table above
5. Map matched fields to data categories for the privacy policy

**What this enables:**
- Auto-detect that the project collects email, name, phone, address
- Know which models store personal data (User, Profile, Customer, etc.)
- Generate accurate "Information We Collect" section with specific data types
- Detect special category data (health, biometric) for heightened GDPR obligations
- Generate accurate data retention sections (if `createdAt`/`updatedAt` present)

### GDPR Implications by Field Type

| Field Pattern | GDPR Implication | Policy Section Affected |
|--------------|------------------|----------------------|
| `email` | Legitimate interest or consent | Legal basis, contact data |
| `password` | Security obligation (Art. 32) | Security measures |
| `dateOfBirth`/`age` | Children's privacy (Art. 8) if under 16 | Children's section, age verification |
| `healthData` | Special category (Art. 9) — needs explicit consent | Special categories, DPIA required |
| `ipAddress` | Personal data per CJEU — legitimate interest basis | Technical data, cookies |
| `biometric` | Special category (Art. 9) — needs explicit consent | Special categories, DPIA required |
| `location`/`address` | May trigger additional location tracking disclosures | Location data section |
| `ssn`/`nationalId` | Sensitive data — strict necessity requirement | Government ID handling |

### Related ORMs to Consider (Future)

- **Drizzle ORM** — `schema.ts` files with TypeScript definitions
- **TypeORM** — Entity decorators in TypeScript classes
- **Sequelize** — Model definitions in JS/TS
- **Mongoose** — Schema definitions for MongoDB

All follow similar patterns: model/entity definitions with field names and types that can be scanned for personal data indicators.

## 7. Recommendations for v5.0

### Output Format Priority

1. **Markdown** (already done) — Keep as default. Lives in repo.
2. **HTML** (new, high priority) — Single self-contained HTML file with inline CSS. Zero dependencies. Professional look.
3. **PDF** (new, lower priority) — Either via pdfmake or "open HTML in browser and print." Can defer to v6.0.

### HTML Output Spec

- Single `.html` file per document (e.g., `PRIVACY_POLICY.html`)
- Inline CSS (~60 lines), no external dependencies
- System font stack: `-apple-system, BlinkMacSystemFont, 'Segoe UI', Roboto, sans-serif`
- Max-width 800px, centered, generous whitespace
- Auto-generated table of contents with anchor links
- Print media query for clean PDF printing from browser
- Meta tags: charset, viewport, title
- Professional, Stripe-like minimal aesthetic

### Smart Scanning: Prisma Schema

- Add new scanner: `src/scanner/prisma.ts`
- Detect `schema.prisma` files
- Parse model/field definitions
- Map field names to personal data categories
- Feed detected categories into privacy policy generator for accurate "data we collect" sections
- Detect special category data for GDPR Art. 9 obligations

### CLI Flag Design

```bash
# Current (markdown only)
codepliant go ./my-project

# New: specify output format
codepliant go ./my-project --format md        # default, current behavior
codepliant go ./my-project --format html      # single HTML files
codepliant go ./my-project --format pdf       # if we implement it
codepliant go ./my-project --format all       # all formats

# Or just always output both MD and HTML
codepliant go ./my-project                    # outputs .md + .html
```
