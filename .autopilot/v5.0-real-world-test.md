# v5.0 Real-World Test Report

**Date:** 2026-03-16
**Tester:** QA Agent (automated)
**Method:** Shallow-cloned 5 real open-source projects, ran `codepliant scan` and `codepliant go` against each.

---

## Executive Summary

Codepliant performs **well on JavaScript/TypeScript monorepos** but has **critical blind spots** for non-JS ecosystems. It detected 0 services for an Elixir project (Plausible) and only 1 of 8+ services for a Ruby/Rails project (Maybe). Even within JS projects, several important service categories are missed (Google APIs, Redis, additional AI SDKs, specific auth providers). The generated documents are structurally sound and professionally formatted, but accuracy depends entirely on detection completeness.

**Overall detection accuracy: ~55-60%** (weighted across projects)

## Critical Issues Found

### P0: No Ruby/Gemfile Support
Maybe Finance is a Ruby on Rails app with Stripe, Plaid, OpenAI, Sentry, Intercom, and AWS S3. Codepliant detected almost nothing. Ruby is the 7th most popular language on GitHub. Rails apps are extremely common in SaaS.

### P0: No Elixir/mix.exs Support
Plausible Analytics got zero detections. Elixir is niche but prominent in web analytics and real-time systems.

### P1: Missing Common Service Patterns
Even in JS/TS projects, these are consistently missed:
1. **Google APIs** (`googleapis`, `@googleapis/*`, `google-auth-library`) -- Cal.com's entire calendar integration missed
2. **Anthropic SDK** (`@anthropic-ai/sdk`) -- AI service with privacy implications
3. **Redis** (`ioredis`, `@upstash/redis`, `redis`, `connect-redis`) -- Session/cache storage
4. **AWS SES** (`@aws-sdk/client-sesv2`) -- Email delivery
5. **AWS Lambda** (`@aws-sdk/client-lambda`) -- Serverless execution
6. **Specific OAuth providers** (`passport-google-oauth20`, `passport-microsoft`, `@node-saml/passport-saml`)
7. **Vercel AI SDK** (`@ai-sdk/*`) -- AI service wrappers
8. **WebAuthn** (`@simplewebauthn/*`) -- Biometric auth data
9. **Cloudflare** -- CDN/proxy (sees all traffic)
10. **Datadog** (`@datadog/*`) -- Monitoring/profiling

### P1: Prisma Personal Data Fields Not Detected
Both Cal.com and Documenso have Prisma schemas with clear User models containing email, name, password, twoFactorSecret, etc. The scan output has no `personalDataFields` key. Either the feature is not implemented or not working.

### P2: Nodemailer False Label for Non-JS Email
When SMTP env vars are found in a non-JS project (Maybe), they get labeled as "nodemailer". Should be labeled generically as "SMTP Email" or detect the actual framework (Action Mailer for Rails).

## Recommendations

### Immediate (before any release)
1. **Add service patterns for:** googleapis, @anthropic-ai/sdk, @ai-sdk/*, redis/ioredis/@upstash/redis, @aws-sdk/client-sesv2, passport-google-oauth20, passport-microsoft, @datadog/*, cloudflare
2. **Fix Prisma personal data field detection** -- this was likely implemented but is not producing output

### Short-term (next sprint)
3. **Add Ruby/Gemfile scanner** -- Parse `gem "name"` lines, map to known services (stripe, sentry-ruby, plaid, etc.)
4. **Add Elixir/mix.exs scanner** -- Parse `{:name, "version"}` tuples, map to known services
5. **Generic SMTP detection** -- When SMTP env vars found without a JS email package, label as "SMTP Email Service" not "nodemailer"

### Medium-term
6. **Add service pattern database expansion** -- The current registry covers ~30-40 services. Real-world projects use 50-100+. Need at least: Datadog, New Relic, Cloudflare, Plaid, various Google Cloud services, Azure services, Twitch/Discord/Slack OAuth
7. **Multi-ecosystem detection in monorepos** -- Twenty has both JS and Go code. A monorepo might have a Rails API + React frontend.
8. **Confidence scoring** -- Instead of binary detected/not-detected, show confidence levels. A type import is less meaningful than an actual API call.

Cal.com's actual privacy policy (at cal.com/privacy) covers:
- Specific mention of Google Calendar, Microsoft Outlook, Apple Calendar integrations
- CalVideo (their video conferencing service)
- Stripe payment processing
- Analytics services (PostHog)
- Cookie usage details
- International data transfers (US/EU)
- Specific retention periods per data type
- Children's privacy (COPPA)
- California residents (CCPA)

**What Codepliant's generated policy is missing vs their real one:**
1. No mention of Google Calendar integration (not detected)
2. No mention of video conferencing data
3. No state-specific sections (CCPA for California)
4. No children's privacy section (COPPA)
5. No international data transfer section
6. Generic retention periods instead of specific ones
7. No mention of calendar/scheduling data as a category

**What Codepliant's policy does well vs theirs:**
1. Better structured GDPR legal basis table
2. More detailed third-party service listing
3. DPA section (separate document)
4. AI Act readiness (for projects that use AI)
